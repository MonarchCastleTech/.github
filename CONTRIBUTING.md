# Contributing

Thank you for improving Monarch Castle Technologies repositories. Check the target repository for more specific instructions; those instructions take precedence over this default guidance.

## Before you open a pull request

1. Create a focused branch from the repository's documented base branch and explain the decision problem the change addresses.
2. Preserve **data provenance**: identify source, collection or observation date, applicable licence or terms, and every material transformation.
3. Run the repository's documented **deterministic validation** command. In the pull request, record the exact command and its result so another contributor can repeat it.
4. For a forward-looking statement, maintain a **forecast record** with its time horizon, probability or uncertainty, assumptions, and independently scoreable outcome. Clearly separate observations, assessments, and forecasts.
5. Write public copy in **English**. Do not introduce comparative, accuracy, scale, customer, or superlative claims without the evidence and approval required by the applicable claims policy.
6. Meet **accessibility** expectations: descriptive link text, meaningful image alt text, heading order, and readable contrast in both GitHub light and dark themes.
7. Add or update a **test** for every behavioral or content-contract change, run the relevant tests locally, and include the results in the pull request.

## Reviewable pull requests

Keep a pull request small, describe the evidence behind changed public copy, and avoid unrelated formatting churn. Never include credentials, private data, or live vulnerability details.
